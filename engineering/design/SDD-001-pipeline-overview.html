§1 · Architecture

System architecture

Written

The pipeline is organised into four horizontal layers. Each layer communicates with the next through a single well-defined contract — no layer reaches past its immediate neighbour. This separation means adapters, processing stages, output serialisers, and client interfaces can all evolve independently.

Color encodes layer: gray = input adapters (format-specific, swappable), purple = core processing (ML models), teal = outputs (standard formats), blue = client interfaces (API consumers).

§2 · Contracts

Inter-stage data contracts

Written

Each handoff between layers is a typed contract. No stage may reach past its immediate neighbour. If a contract changes, only the two stages on either side are affected.

Adapter → Ingest

Contract
path: str # absolute path to a local, readable audio file

Adapters are responsible for producing a path. They handle all network I/O, device capture, format detection, and temp-file management. Ingest does not know and does not care how the file arrived.

Ingest → Separate

Contract
np.ndarray shape=(2, N) dtype=float32 values∈[-1.0, 1.0] sr=44100

A stereo float32 array at 44,100 Hz. This is the universal internal audio representation. Separate never reads a file — it always receives this array from Ingest.

Separate → Transcribe

Contract
dict[str, np.ndarray] # keys: "drums" | "bass" | "vocals" | "other"
# each value: shape=(2, N), dtype=float32, sr=44100

One stereo array per stem. Transcribe receives each stem independently and runs Basic Pitch on each. The drum stem is routed to onset detection, not pitch transcription (see ADR-003, pending).

Transcribe → Outputs

Contract
dict[str, pretty_midi.PrettyMIDI] # one object per pitched stem

One PrettyMIDI object per stem. Outputs serialise these to .mid files, and pass them to music21 for score generation. Stems are written to FLAC directly from the Separate output.

Pipeline → API

Contract
JobResult { stems: dict[str, Path], midi: dict[str, Path], score: dict[str, Path] }

The pipeline runner returns a JobResult with file paths for all produced artifacts. The API layer reads these paths and serves the files to clients. It does not re-process anything.

Layer 1 · Input adapters

Adapters are thin wrappers with one job: produce a path to a local audio file. The pipeline below never knows or cares how the audio arrived. Adding a new source is zero-impact on all downstream stages.

Adapter · File

File adapter

Written

The simplest adapter. Accepts a local filesystem path, validates it, and returns it unchanged. All format detection and decoding is deferred to the Ingest stage via librosa.

Supported formats

FormatExtensionsCodecBackend
MP3.mp3MPEG Layer III (lossy)audioread + FFmpeg
WAV.wavPCM 8/16/24/32-bitsoundfile (native)
FLAC.flacFree Losslesssoundfile (native)
AAC.aac, .m4aAdvanced Audio Coding (lossy)audioread + FFmpeg
OGG Vorbis.oggVorbis (lossy)soundfile (native)
AIFF.aiff, .aifPCM (lossless)soundfile (native)
MP4 audio.mp4AAC or ALACaudioread + FFmpeg
Dependency risk

MP3 and MP4 require FFmpeg in $PATH. Absence causes audioread.NoBackendError at runtime. The deployment image must install FFmpeg explicitly.

Implementation
from pathlib import Path

SUPPORTED = {'.mp3','.wav','.flac','.aac','.m4a','.ogg','.aiff','.aif','.mp4'}

def file_adapter(path: str) -> str:
    p = Path(path)
    if not p.exists():
        raise AdapterError(f"File not found: {path}")
    if not p.is_file():
        raise AdapterError(f"Path is not a file: {path}")
    if p.suffix.lower() not in SUPPORTED:
        raise AdapterError(f"Unsupported format: {p.suffix}")
    return str(p.resolve())
Adapter · URL

URL adapter

Pending design

Accepts a URL, downloads or extracts audio to a temp file, and returns the local path. The pipeline never touches a network socket — that is entirely this adapter's responsibility.

Planned sources

YouTube / SoundCloud / Spotify

Use yt-dlp for YouTube and SoundCloud. Spotify requires a separate legal/auth consideration — flagged as open question.

Direct audio URLs

Any direct link ending in a supported extension (.mp3, .wav, .flac etc). Use httpx with streaming download, size-limited to prevent abuse.

Open question · Q-URL-1

What is the maximum file size / duration we accept from URL inputs? This affects temp disk usage and Demucs inference time. Needs a product decision before implementation.

Open question · Q-URL-2

Spotify streams require OAuth and are DRM-protected. Do we support Spotify URLs at all, or require users to upload local files for that source?

Adapter · Stream

Stream adapter

TBD

Accepts a live audio stream over WebSocket, buffers it to a temp file, and passes the path to the pipeline once a sufficient window has been captured. This adapter is the most architecturally complex because the pipeline is currently batch-oriented — it processes a complete file, not a rolling buffer.

Open question · Q-STREAM-1

Do we buffer the entire stream before processing (latency: full duration) or do we process in rolling windows (latency: window size, quality: lower)? This is a fundamental architecture decision. Rolling windows require a different pipeline variant — Demucs is not streaming-native.

Open question · Q-STREAM-2

What audio codec does the WebSocket stream carry? Raw PCM, Opus, or something else? The adapter needs to handle codec negotiation or enforce a single format.

Recommendation

For v1, support stream buffering (capture full stream → write temp file → process as batch). Rolling-window streaming can be a v2 feature with a separate pipeline variant. This avoids blocking the batch pipeline design on streaming architecture decisions.

Adapter · Device

Device adapter

TBD

Captures audio from a microphone or line-in device, writes it to a temp file, and hands the path to the pipeline. Intended for instrument-to-score use cases — a musician plays into a microphone and receives a score of what they played.

Open question · Q-DEV-1

Is device capture a server-side concern (capturing from a server microphone) or a client-side concern (browser captures audio via MediaRecorder API, then uploads)? Almost certainly client-side — which means this adapter is really a variant of the Stream or File adapter on the server, and the capture logic lives in the web embed SDK.

Recommendation

The web embed handles MediaRecorder capture and sends the resulting blob to the API as a file upload. The server-side adapter for this case is just the File adapter receiving an uploaded temp file. No separate server-side device adapter is needed.

Layer 2 · Core processing pipeline

Three sequential stages. Each stage is stateless — it receives a well-typed input, performs computation, and returns a well-typed output. No stage writes to disk; the pipeline runner handles all persistence.

Stage 1 · Ingest

Ingest

Written — see SDD-002

→

→

Ensure stereo

Fully documented in SDD-002. Decodes any supported codec, resamples to 44,100 Hz, converts to float32, and upmixes mono to stereo. Returns an np.ndarray of shape (2, N).

Key decisions (from SDD-002)

44,100 Hz target rate matches Demucs training data. float32 in [-1,1] is the universal ML internal format. Mono upmix duplicates the single channel. Clipping is detected and logged but not repaired.

Stage 2 · Separate

Separate (Demucs)

Pending — SDD-003

What Demucs does

Demucs (htdemucs variant) is a hybrid neural network trained on the MUSDB18 dataset of professionally mixed music. It processes audio simultaneously in the waveform domain and the spectrogram domain — combining what both representations reveal about instrument identity — and produces four isolated stems. Each stem is a stereo float32 array at 44,100 Hz, same format as the input.

The four stems

StemContainsTranscription path
drumsKick, snare, hi-hat, cymbals, tomsOnset detection (not pitch — see ADR-003)
bassBass guitar, synth bassBasic Pitch → MIDI
vocalsLead and backing vocalsBasic Pitch → MIDI (melody)
otherGuitar, piano, strings, all elseBasic Pitch → MIDI

Performance profile

Hardware4-min trackNotes
CPU (modern)~8–12 minToo slow for synchronous API response
GPU (T4)~25–40 secAcceptable for async job queue
GPU (A10G)~10–15 secGood for near-real-time
Risk · GPU dependency

Demucs on CPU is too slow for production use. GPU instances are required. This has significant infrastructure cost implications — see ADR-002 (pending) for GPU provisioning strategy.

Implementation sketch
from demucs.pretrained import get_model
from demucs.apply import apply_model
import torch, numpy as np

def separate(audio: np.ndarray, device: str = "cuda") -> dict[str, np.ndarray]:
    model = get_model("htdemucs")
    model.to(device)
    tensor = torch.tensor(audio).unsqueeze(0)   # (1, 2, N)
    with torch.no_grad():
        sources = apply_model(model, tensor)[0]  # (4, 2, N)
    return {
        name: sources[i].cpu().numpy()
        for i, name in enumerate(model.sources)
        # model.sources = ["drums", "bass", "other", "vocals"]
    }
Open question · Q-SEP-1

Should the model be loaded once at worker startup (singleton, fast per-job) or loaded per job (slow, but safe for multi-process workers)? Singleton is the right answer for GPU workers but requires careful memory management.

Open question · Q-SEP-2

Demucs produces an "other" stem that is a catch-all. For a song with piano, guitar, and strings, all three are mixed into one stem and then transcribed together. Is that acceptable quality, or do we need a more fine-grained separation model?

Stage 3 · Transcribe

Transcribe (Basic Pitch)

Pending — SDD-004

What Basic Pitch does

Basic Pitch (Spotify Research) is a neural network trained to perform polyphonic pitch detection — it can identify multiple simultaneous notes in an audio stream. Unlike classical methods like pYIN, it handles chords and layered notes. It accepts a mono or stereo audio array and returns a PrettyMIDI object containing detected notes with pitch, onset time, offset time, and velocity.

Running per-stem, not on the full mix

A key architectural decision: we run Basic Pitch on each stem independently, not on the full mix. A stem contains a much narrower range of timbres, which means fewer competing pitches for the model to disambiguate. This produces materially better MIDI quality at the cost of an extra Demucs step.

Key parameters

ParameterDefaultEffect
onset_threshold0.5Minimum confidence to accept a note onset. Higher = fewer false notes.
frame_threshold0.3Minimum confidence per frame to sustain a note.
minimum_note_length58msFilters out very short events that are likely detection artefacts.
minimum_frequency32.7 Hz (C1)Lower bound for pitch detection.
maximum_frequency2093 Hz (C7)Upper bound. Raise for piccolo/violin, leave for most instruments.
Open question · Q-TRX-1

Should thresholds be fixed per deployment, configurable per job, or tuned per stem type (e.g. lower onset threshold for bass, higher for vocals)? Per-stem tuning likely improves quality but adds complexity.

Open question · Q-TRX-2

The drum stem has no stable pitch. What handles drum transcription? Options: (a) skip it entirely for now, (b) use onset detection only (beat times, no pitch), (c) use a dedicated drum transcription model like ADTLib. This is ADR-003.

Implementation sketch
from basic_pitch.inference import predict
from basic_pitch import ICASSP_2022_MODEL_PATH
import pretty_midi, numpy as np

def transcribe(stem: np.ndarray, sr: int = 44100) -> pretty_midi.PrettyMIDI:
    # Basic Pitch expects mono; mix down stereo stem
    mono = stem.mean(axis=0) if stem.ndim == 2 else stem
    _, midi_data, _ = predict(
        mono,
        ICASSP_2022_MODEL_PATH,
        onset_threshold=0.5,
        frame_threshold=0.3,
        minimum_note_length=58,
    )
    return midi_data

def transcribe_all(stems: dict[str, np.ndarray]) -> dict[str, pretty_midi.PrettyMIDI]:
    pitched = {k: v for k, v in stems.items() if k != "drums"}
    return {name: transcribe(audio) for name, audio in pitched.items()}
Layer 3 · Outputs

Outputs are serialisation of the pipeline's results into open standard formats. No processing happens here — only writing well-typed data to files that downstream tools and clients can consume without custom parsers.

Output · Stems

Stems

Pending

Each stem array from the Separate stage is written to disk as a FLAC file. FLAC is lossless and compressed — it preserves the full float32 precision of the Demucs output while being significantly smaller than raw WAV.

Output files per job

FileContentTypical size (4-min track)
drums.flacIsolated drum stem~25–40 MB
bass.flacIsolated bass stem~20–35 MB
vocals.flacIsolated vocal stem~20–35 MB
other.flacAll remaining instruments~25–45 MB
Serialisation
import soundfile as sf
from pathlib import Path

def write_stems(stems: dict[str, np.ndarray], output_dir: Path, sr: int = 44100):
    for name, audio in stems.items():
        path = output_dir / f"{name}.flac"
        # soundfile expects (samples, channels); our arrays are (channels, samples)
        sf.write(str(path), audio.T, sr, subtype='PCM_24')
Open question · Q-STEM-1

Should stems be stored permanently or treated as ephemeral job artifacts with a TTL (e.g. 24 hours)? Storage cost for a high-volume service is non-trivial. Needs a product and infrastructure decision.

Output · MIDI

MIDI

Pending

Each PrettyMIDI object from the Transcribe stage is written to a standard .mid file. MIDI is a binary format encoding note events — pitch, velocity, onset time, offset time — and is readable by every DAW, notation tool, and music software.

What's in a MIDI file

A .mid file from this pipeline contains one instrument track per stem. Each note event encodes: MIDI note number (0–127, where 60 = middle C), start time in seconds, end time in seconds, and velocity (0–127, encoding how hard the note was played). Tempo and time signature are estimated by pretty_midi from the note density.

Serialisation
from pathlib import Path
import pretty_midi

def write_midi(midi_map: dict[str, pretty_midi.PrettyMIDI], output_dir: Path):
    for name, midi in midi_map.items():
        path = output_dir / f"{name}.mid"
        midi.write(str(path))
Open question · Q-MIDI-1

Should the MIDI output assign General MIDI program numbers (instrument patches) to each stem? E.g. bass → program 33 (Electric Bass). This makes the MIDI file sound more realistic in a DAW but requires a mapping table and adds a layer of assumption.

Output · Score

Score (MusicXML / PDF)

Pending — SDD-005

The score output converts PrettyMIDI note events into human-readable notation via music21. music21 understands music theory — key signatures, time signatures, rhythmic quantisation, beaming — and outputs MusicXML, which is the open standard for notation interchange. MusicXML is then rendered to PDF by LilyPond or MuseScore CLI, both of which run headless.

The quantisation problem

Basic Pitch produces note times in seconds with floating-point precision. A real score needs notes quantised to a rhythmic grid (quarter notes, eighth notes, etc.). music21's quantisation is configurable but imperfect — expressive timing (rubato, swing) will be forced onto a strict grid, which may look odd for certain genres. This is a known limitation. See open question below.

Implementation sketch
from music21 import converter, stream, tempo
import pretty_midi, tempfile
from pathlib import Path

def midi_to_score(midi: pretty_midi.PrettyMIDI, output_dir: Path, stem_name: str):
    # Write temp .mid, read back via music21
    tmp = tempfile.NamedTemporaryFile(suffix='.mid', delete=False)
    midi.write(tmp.name)

    score = converter.parse(tmp.name)
    score = score.quantize(quarterLengthDivisors=(4, 3))  # 16th note grid + triplets

    # MusicXML
    xml_path = output_dir / f"{stem_name}.musicxml"
    score.write('musicxml', fp=str(xml_path))

    # PDF via MuseScore (must be in PATH)
    pdf_path = output_dir / f"{stem_name}.pdf"
    score.write('musicxml.pdf', fp=str(pdf_path))
Open question · Q-SCORE-1

Should score quantisation be strict (fixed grid, clean notation) or attempt to preserve expressive timing? Strict is better for classical/structured music; expressive preservation matters for jazz/blues. Needs a product decision — likely a configurable option. See ADR-004.

Open question · Q-SCORE-2

LilyPond produces higher-quality engraving than MuseScore CLI but is less widely available as a system package. Do we ship LilyPond in our container image, use MuseScore, or support both?

Layer 4 · Clients & API

The API is the boundary between the pipeline and the outside world. All clients — web embeds, mobile SDKs, third-party integrations — speak to the same REST + WebSocket API. The API never re-processes; it only manages jobs and serves artifacts.

All clients are thin API wrappers by design. No ML logic runs on client devices. Demucs requires a GPU; model weights are ~80MB; the pipeline must serve iOS, Android, web, Spotify, and Apple Music from a single implementation. The API-first approach is the only architecture that satisfies all of these constraints simultaneously. The mental model: this is exactly how game backend services work — Unity doesn't run matchmaking on-device, it calls an API. Auris Vive clients are the same. See ADR-006 for the full decision, all four options considered, and the on-device future consideration.

Note for mobile SDK implementors: artifact downloads are binary files (FLAC stems, MIDI, MusicXML), not just JSON — the SDK must handle chunked binary downloads with progress reporting, which differs from a standard REST client pattern.

Client · Web

Web embed

TBD

A JavaScript SDK that web applications embed to submit audio and receive results. Wraps the REST and WebSocket API. Handles file upload progress, job polling or WebSocket subscription, and result download.

Open question · Q-WEB-1

Is the web embed a drop-in <script> tag with a UI widget, or a headless JS module that applications wire into their own UI? Open question · Q-WEB-2

For device capture (microphone → pipeline), does the web embed use MediaRecorder API to capture audio in the browser, then upload as a file? What codec does the browser output? (Typically Opus in a WebM container — the File adapter needs to handle this.)

Client · Mobile

Mobile SDK

TBD

Native SDKs for iOS (Swift) and Android (Kotlin) that wrap the REST + WebSocket API. The mobile SDK has the same responsibilities as the web embed — file submission, job tracking, result retrieval — with platform-appropriate patterns (async/await, Combine, Coroutines).

Open question · Q-MOB-1

Do we build native iOS and Android SDKs, a React Native module, or a Flutter plugin? Depends on whether the primary mobile consumer is a native app or a cross-platform one. Needs a product decision.

Note

All audio processing runs server-side. The mobile SDK does not run Demucs or Basic Pitch on-device. It is a thin API client. On-device processing is a potential future direction but is outside scope for v1.
